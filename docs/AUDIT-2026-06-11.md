# تقرير تحليل نظام Sellpert الشامل وخطة العمل

**التاريخ:** 2026-06-11
**المنهجية:** تدقيق متعدد الوكلاء (10 وكلاء): 6 وكلاء متخصصين بالتوازي (معمارية، خطأ الرفع، قاعدة البيانات الحية، Edge Functions، أمان، جودة) + 4 وكلاء تحقق عدائي حاولوا **دحض** كل سبب مرشح لخطأ الرفع بإعادة قراءة الكود وفحص قاعدة البيانات الحية مباشرة (pg_policies، محاكاة RLS، السجلات، المستشارين الأمنيين).

## 1. خريطة النظام

| المكوّن | الواقع |
|---|---|
| الواجهة | React 18 + Vite + TypeScript (strict)، ~103 ملف، SPA بدون مكتبة توجيه — توجيه يدوي مكرر في `App.tsx` و `AdminPanel.tsx` |
| الاستضافة | Vercel (rewrite الكل إلى index.html) |
| الخلفية | Supabase `urdyzbsukcuibadlaath` (ap-southeast-1، Postgres 17): 45 جدولاً، 61 ترحيلاً، 16 دالة Edge محلياً + 3 منشورة غير موجودة في المستودع (queue-worker, ai-chat, respondly-webhook) |
| الأدوار | عمود `merchants.role` (merchant / admin / employee) — ليست في JWT؛ كل سياسة RLS تبحث بالبريد. الصلاحيات مصفوفة jsonb |
| الوكيل (proxy) | `sellpert-proxy/proxy.js` — تمرير نداءات Trendyol API فقط |
| تدفق الاستيراد | **لا يستخدم Storage إطلاقاً**: يفك xlsx/csv/zip في المتصفح (`platformParsers.ts`) ثم يكتب مباشرة في +10 جداول مالية بتوكن المستخدم نفسه (تخضع لـ RLS) |
| الإدخال اليدوي | عبر دالة Edge ‏`manual-entry` بمفتاح service-role (يتجاوز RLS) — لهذا يعمل للموظفين بينما يفشل استيراد الملفات |
| Storage | حاوية واحدة `merchant-assets` (عامة، 2MB، صور فقط) — شعارات التجار فقط |

**أرواح ميتة / انجراف:** `EmployeePanel.tsx` (42KB) مستورد ولا يُعرض أبداً؛ `dashboard.html` و`login.html` قديمة (login يدمج مفتاح anon والرابط الحي)؛ مجلدا `{css,js}` فارغان؛ `supabase-schema.sql` يصف جدولين بينما القاعدة الحية فيها 45؛ ترحيل أزال super_admin لكن ~20 سياسة وis_admin() ما زالت تشير إليه؛ نظامان غير متوافقين للموظفين (Team للتاجر يخزن الصلاحيات ككائن، EmployeesView للإدارة يخزنها كمصفوفة — والواجهة لا تقرأ إلا المصفوفة).

## 2. خطأ رفع الملفات للموظفين — السبب الجذري (مؤكد)

> **الخلاصة:** المشكلة ليست في Storage ولا في الملفات نفسها. السبب سلسلة من ثلاث عوائق، اثنان في سياسات RLS وواحد في الكود. **الحل الجاهز:** [docs/fix-employee-uploads.sql](fix-employee-uploads.sql)

### السلسلة المسببة بالترتيب الذي يصطدم بها الموظف

1. **(العائق الأول — المؤكد بمحاكاة RLS حية)** الموظف لا يستطيع قراءة جدول `merchants`: السياسات الموجودة هي `merchant_select_own` (صفّه هو فقط) و `admin_all_merchants` (عبر `is_admin()` التي تقبل admin/super_admin فقط). النتيجة: القائمة المنسدلة لاختيار التاجر في شاشة الاستيراد **فارغة** (`AdminPanel.tsx:284` ← `ImportFilesView.tsx:674`)، فيظهر جدار «⚠️ اختر التاجر أولاً قبل رفع الملفات» ومنطقة الرفع لا تُعرض أصلاً (`ImportFilesView.tsx:711`).

2. **(العائق الثاني — كامن خلف الأول)** سياسات الكتابة على كل جداول الاستيراد admin-only: `platform_file_uploads`, `orders`, `inventory`, `account_transactions`, `ad_metrics`, `returns`, `goods_received`, `product_performance_snapshots`, `inbound_shipments`, `inbound_shipment_items`. أي إدراج بتوكن موظف يُرفض بـ `42501 new row violates row-level security policy`. الجداول الوحيدة الموسّعة سابقاً لتشمل employee هي `products`, `platform_deals`, `entry_sessions` — دليل أن ترحيل صلاحيات الموظف بدأ ولم يكتمل.

3. **(عائق ثالث — خطأ كود كامن)** `ImportFilesView.tsx:594-599` يتجاهل خطأ إدراج سجل التدقيق في `platform_file_uploads` ويُكمل بـ `uploadId = ''`، فتفشل حتى الجداول المسموح بها للموظف بخطأ `22P02 invalid input syntax for type uuid: ""`، ولا يبقى أي أثر للفشل في سجل التدقيق.

### أدلة من بيئة الإنتاج

- القاعدة الحية فيها **صفر** حسابات `role='employee'` حالياً (5 admin، 4 merchant): حسابات الطاقم الثلاثة (E-6752, E-1541, E-6631 — البادئة E تثبت أنها أُنشئت كموظفين) **رُفعت كلها إلى admin كحل التفافي** — وهذا يعني أن كل موظف يملك اليوم صلاحيات كاملة (حذف تجار، `wipe_merchant_data`، إدارة اشتراكات) بغض النظر عن قسمه.
- كل سجلات `platform_file_uploads` الـ 25 نجاحات لمشرفين؛ لا يوجد سجل فشل واحد لأن محاولة الموظف تموت قبل أن يُكتب أي سجل.
- سجل رفع واحد عالق في `processing` منذ 2026-06-04.
- المقارنة الدامغة: دالة `manual-entry` تسمح صراحة بـ employee وتكتب بمفتاح service-role — لهذا «الإدخال اليدوي» كان يعمل بينما «استيراد الملفات» يفشل.

### الإصلاح (جاهز للتطبيق)

**أ. قاعدة البيانات** — ملف [fix-employee-uploads.sql](fix-employee-uploads.sql):
دالة `is_staff()` بنمط SECURITY DEFINER (إجباري — سياسة على merchants تستعلم merchants مباشرة تنفجر بـ 42P17 infinite recursion) + سياسة قراءة merchants للطاقم + توسيع سياسات الكتابة العشر. مع خيار تشدد يربط كتابة الموظف بصلاحية `upload_files`.

**ب. الكود** — `src/pages/admin/ImportFilesView.tsx:594`:

```ts
const { data: audit, error: auditErr } = await supabase
  .from('platform_file_uploads')
  .insert({ ... })
  .select('id')
  .single()
if (auditErr || !audit?.id) {
  allErrors.push(`${entry.file.name}: فشل إنشاء سجل الرفع — ${auditErr?.message ?? 'صلاحيات غير كافية'}`)
  setFiles(p => p.map(f => f.id === entry.id ? { ...f, stage: 'failed', progress: 0 } : f))
  continue
}
const uploadId = audit.id
```

**ج. بعد التحقق:** إعادة حسابات الطاقم الثلاثة إلى `role='employee'` لاستعادة نظام الصلاحيات.

**د. خيار معماري أفضل على المدى المتوسط:** نقل حفظ الاستيراد كاملاً إلى دالة Edge بمفتاح service-role (كنمط manual-entry) — يقلّص سطح RLS إلى نقطة واحدة، يصمد أمام إغلاق التبويب منتصف الحفظ، ويغني عن منح الموظفين كتابة مباشرة على 10 جداول مالية.

### متوسطة

- سياسات INSERT دائمة الصحة على `audit_log` و`ai_insights` — أي عميل يحقن سجلات تدقيق مزيفة.
- `sellpert-proxy` يفشل مفتوحاً إن لم يُضبط `PROXY_SECRET`؛ `create-merchant` يقبل role من جسم الطلب كما هو.
- `vercel.json` بلا أي ترويسات أمنية (CSP, HSTS, X-Frame-Options...).
- CORS = `*` في كل الدوال الـ 16 + رسائل الأخطاء الخام (وأجسام ردود منصات خارجية) تُعاد للمتصفح.
- `manual-entry`: upsert بدون إزالة تكرار مفتاح التعارض — إدخال يدوي بمنتجين لنفس (تاجر/منصة/يوم) ينفجر بـ 500 «ON CONFLICT cannot affect row a second time» — **هذا أيضاً «خطأ يظهر للموظفين»** في مسار الإدخال اليدوي.

## 4. الموثوقية والجودة

1. **(حرج) ابتلاع أخطاء Supabase**: ‏25 موضعاً بنمط `const { data } = await supabase...` بلا قراءة `error`، **صفر** console.error في كل src، نظام Toast مستخدم في 13 ملفاً فقط. هذا النمط هو ما جعل خطأ الموظفين «غامضاً» أصلاً.
2. **(عالي) صفر اختبارات، صفر CI، صفر lint** على تطبيق إنتاجي مالي +9000 سطر. أخطر ملف (`platformParsers.ts` — 1096 سطراً من دوال نقية تغذي جداول مالية) قابل للاختبار فوراً، وملف العينة `seller-1148158-sales-report-*.xlsx` موجود كـ fixture جاهز.
3. **(عالي — خطأ صحة بيانات كامن)** ‏82 استدعاء `select('*')` بلا حدود، وPostgREST يقتطع صامتاً عند 1000 صف: لوحة التحكم وكشف الحساب يحسبان KPIs في المتصفح — **بمجرد تجاوز تاجر 1000 صف ستصبح الأرقام خاطئة بدون أي خطأ ظاهر**.
4. **(عالي)** حزمة الدخول 867KB لأن Dashboard (وبالتالي recharts) مستورد eager — حتى صفحة تسجيل الدخول تحمّله؛ AdminPanel يستورد 24 شاشة إدارية دفعة واحدة (459KB + سلسلة xlsx ‏424KB).
5. **(متوسط)** 5 ملفات وحشية (770-1182 سطراً) تخلط محرك حفظ + قوالب تقارير + 8-10 مكونات داخلية؛ لا طبقة بيانات مشتركة (صفحة المنتجات تطلق ~11 استعلاماً لكل زيارة، و464 useState عبر الصفحات)؛ 223 استخدام `any`؛ أنواع الجداول مكتوبة يدوياً ومنجرفة عن المخطط الفعلي.
6. **(متوسط)** حلقات كتابة N+1 (`AdminProductsView.tsx:54`, `QuickInventory.tsx:77`) — طلب لكل صف بدل upsert واحد.
7. **(منخفض)** أداء RLS: ‏225 تحذير سياسات متعددة + 77 تحذير `auth_rls_initplan` (إعادة تقييم auth.email() لكل صف — ملموس على ad_metrics ذات 39,882 صفاً)؛ 7 مفاتيح أجنبية بلا فهارس، 20 فهرساً غير مستخدم؛ pg_cron يطرق كل دقيقة طابوراً فارغاً ويغرق السجلات.

## 5. خطة العمل المرحلية

### المرحلة 0 — اليوم (إصلاح خطأ الموظفين)
1. تطبيق [fix-employee-uploads.sql](fix-employee-uploads.sql) على القاعدة (دالة is_staff + سياسة قراءة merchants + توسيع 10 سياسات كتابة).
2. إصلاح ابتلاع خطأ سجل التدقيق في `ImportFilesView.tsx:594` (الكود في §2-ب) + نشر.
3. إصلاح dedupe في `manual-entry` (تجميع الصفوف المتشاركة في مفتاح التعارض قبل upsert) — يصلح خطأ الإدخال اليدوي متعدد المنتجات.
4. التحقق: إنشاء موظف تجريبي بصلاحية upload_files ← يرى قائمة التجار ← يرفع ملفاً ← يُحفظ. ثم إعادة حسابات الطاقم الثلاثة إلى employee.
5. تعليم سجل الرفع العالق منذ 2026-06-04 كـ failed.
